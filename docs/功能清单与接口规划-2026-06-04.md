## 1. 文档目的

这份文档用于把当前网站拆成一份更适合产品、前端、后端协作的功能清单，重点回答 3 件事：

- 每个功能应该在哪个页面或模块呈现
- 每个功能当前需要哪些数据对象
- 如果后续接真实后端，需要什么接口

本文以当前站点实现为基础，兼顾后续可落地的服务端规划。

## 4. 按页面拆解功能、数据和接口

## 4.1 首页 / 登录页 `/`

### 页面应呈现的功能

- 品牌首页视觉
- 登录表单
- 注册入口
- 忘记密码入口

### 前端需要的数据

- 品牌文案
- 登录错误提示
- 注册成功提示
- 找回密码提示

### 建议接口

#### 1. 登录

- `POST /api/auth/login`

请求体：

```json
{
  "account": "linche@example.com",
  "password": "******"
}
```

返回：

```json
{
  "user": {
    "id": "user_001",
    "nickname": "林澈",
    "email": "linche@example.com",
    "membership": "Plus",
    "balance": 168.5
  },
  "token": "jwt_or_session_token"
}
```

#### 2. 注册

- `POST /api/auth/register`

#### 3. 找回密码

- `POST /api/auth/reset-password`

## 4.2 用户主页 `/home`

### 页面应呈现的功能

- 用户信息卡片
- 昵称编辑

- 工作台快捷入口
- 记忆体列表

- 当前记忆体 ID

- 分享统计数据

```json
    "visits": 12,
    "activatedUsers": 5,
    "effectiveUsers": 3
  },
  "latestReviewSubmission": {
    "id": "review_001",
    "status": "pending",
    "title": "发帖征集",
    "link": "https://..."
  }

```

- `PATCH /api/user/profile`

请求体：

```json
{
  "nickname": "新的昵称"
```


## 4.3 提示词调试 `/prompt-debug`

### 页面应呈现的功能

- 当前记忆体信息
- prompt 各字段编辑
- 保存草稿
- 测试对话
- 模板导入

### 前端需要的数据

- 当前记忆体的 prompt 草稿
- 可用模板列表
- 模板审核状态
- 测试对话记录

### 建议接口

#### 1. 获取当前记忆体 prompt

- `GET /api/memories/:memoryId/prompt-draft`

#### 2. 保存 prompt 草稿

- `PUT /api/memories/:memoryId/prompt-draft`

#### 3. 发送测试消息

- `POST /api/prompt-debug/chat`

请求体：

```json
{
  "memoryId": "memory-home",
  "message": "今天有点累",
  "draft": {
    "tone": "温和、稳定",
    "personality": "耐心"
  }
}
```

#### 4. 获取模板列表

- `GET /api/prompt-templates`

#### 5. 导入模板到记忆体

- `POST /api/prompt-templates/:templateId/import`

## 4.4 记忆查看 `/memory`

### 页面应呈现的功能

- 月历视图
- 月份切换
- 某日记忆内容
- 空态提示

### 前端需要的数据

- 当前记忆体 ID
- 当前月份的记忆索引
- 日期对应的记忆条目

### 建议接口

#### 1. 获取某记忆体某月份的记忆日历

- `GET /api/memories/:memoryId/calendar?month=2026-06`

返回建议：

```json
{
  "month": "2026-06",
  "days": [
    {
      "date": "2026-06-04",
      "count": 3
    }
  ]
}
```

#### 2. 获取某日记忆详情

- `GET /api/memories/:memoryId/items?date=2026-06-04`

## 4.5 收件箱 `/inbox`

### 页面应呈现的功能

- 消息列表
- 消息详情
- 未读数
- 点击自动已读
- 搜索
- 按类型筛选
- 全部标为已读
- 投稿审核消息同步显示

### 前端需要的数据

- 系统消息列表
- 审核类消息列表
- 未读状态
- 分类
- 搜索关键词

### 建议接口

#### 1. 获取消息列表

- `GET /api/inbox/messages`

支持参数：

- `category`
- `keyword`
- `status=unread`
- `page`
- `pageSize`

#### 2. 标记单条已读

- `POST /api/inbox/messages/:messageId/read`

#### 3. 批量标记已读

- `POST /api/inbox/messages/read-all`

请求体：

```json
{
  "messageIds": ["msg_001", "msg_002"]
}
```

#### 4. 审核进度消息来源

审核消息建议不要单独在前端拼接，后续由后端直接统一下发进消息列表。

也可以保留一层服务端聚合：

- `GET /api/inbox/messages`

由服务端将：

- 系统通知
- 优惠券
- 邀请奖励
- 投稿审核进度

统一合并返回。

```json
{
```

- `GET /api/campaigns/invite`

- `POST /api/campaigns/invite/bind`

请求体：

- `GET /api/campaigns/share/current?memoryId=memory-home`

#### 8. 复制分享链接前的展示数据

- `GET /api/campaigns/share/:code/stats`

## 4.7 分享落地页 `/share/[code]`

### 页面应呈现的功能

- 智能体介绍
- 开始体验按钮
- mock 对话或真实对话入口
- 分享统计自动累积

### 前端需要的数据

- 分享码对应的记忆体信息
- 分享链接状态
- 当前访问者标识

### 建议接口

#### 1. 获取分享页内容

- `GET /api/share/:code`

返回建议：

```json
{
  "code": "SOUL-HOME-2026",
  "memory": {
    "id": "memory-home",
    "name": "日常陪伴",
    "description": "记录生活节奏..."
  }
}
```

#### 2. 记录访问

- `POST /api/share/:code/visit`

返回建议：

```json
{
  "visitorId": "visitor_xxx"
}
```

#### 3. 记录开始体验

- `POST /api/share/:code/activate`

请求体：

```json
{
  "visitorId": "visitor_xxx"
}
```

#### 4. 记录有效使用

- `POST /api/share/:code/effective-use`

请求体：

```json
{
  "visitorId": "visitor_xxx",
  "messageCount": 3
}
```

建议后端定义“有效使用”规则，例如：

- 发出首条消息
- 累计 3 条消息
- 会话持续超过 60 秒

三者选一，避免前后端口径不一致。

## 6. 当前前端已具备但后端未接入的能力

- 首页登录、注册、找回密码流程
- 用户主页昵称编辑
- 记忆体切换
- 投稿提交后同步到主页和收件箱
- 创作分享码自动计数
- 收件箱已读、搜索、筛选、全部已读
- 支付确认弹窗

这些能力现在都靠本地状态模拟，后续只要把接口接上，页面结构本身不需要大改。

## 9. 文档结论

如果把这个站点作为下一阶段开发基础，最适合的协作方式是：

- 前端继续保持当前页面结构
- 后端按本文的对象和接口拆分服务
- 收件箱、分享、审核、充值优先统一成服务端真实数据

这样能保证“心迹档案”当前已经做出来的使用链路不被推翻，只是从本地 mock 平滑升级到真实业务系统。
